### The Three Moats
1. **Physical moat**: Hardware is bolted in. Competitors need a truck and a signed change order to displace us

### Build Sequencing Rule
Always prioritize features that: (a) deliver immediate visible value to already-installed properties at near-zero marginal cost, (b) create a new revenue stream for the property, or (c) strengthen the data layer for AI. Never build features that require displacing existing hardware or that only serve uninstalled prospects.

GateGuard is not building features that use AI. GateGuard is building **an AI engine** that uses features.

### Non-Negotiable Principles

- **Cost discipline**: always benchmark model cost vs output quality. Use Haiku where Haiku is enough. Reserve Opus for tasks that genuinely need it.

## DEALER HIERARCHY (5 tiers вҖ” data isolated per tier)

```
GateGuard Corporate (SO вҖ” System Operator)
  в””в”Җв”Җ MSO (Multi-Site Operator) вҖ” large dealer group, multiple markets
        в””в”Җв”Җ Dealer вҖ” core user, runs their day-to-day business
              в””в”Җв”Җ Sub-Dealer вҖ” authorized reseller under a dealer
                    в””в”Җв”Җ Sales Rep / Independent Rep вҖ” commission-based, may have sub-reps
                          в””в”Җв”Җ Client (Property Manager) вҖ” end customer, read-only portal
```

**What each tier sees:**
- **GG Corporate**: All dealers, all clients, total MRR/ARR, rep network, commissions, dealer scorecards, co-op pool, full KB/product library
- **MSO**: Their child dealers, roll-up MRR dashboard, cross-site WO summary, all client properties, their rep network, bulk pricing tools
- **Dealer**: Their clients only, their sub-dealers, their reps + commissions, quotes + WOs, invoices + MRR, cameras for their accounts, tech tool access codes, KB + manuals, renewal tracker
- **Sub-Dealer**: Their clients only, their own WOs + quotes, their own reps, limited billing view, tech tool access. Cannot see parent dealer's clients.
- **Sales Rep**: Their pipeline only, commissions earned, their sub-reps, quote builder, client contact info. No billing/MRR, no camera access.
- **Client (Property Manager)**: Their properties only, open service tickets, approve quotes, view invoices + pay, camera clips, access logs, device health status.

**Key principle**: All data isolation enforced via Supabase Row-Level Security by org ID. Same database, filtered views вҖ” no custom builds per customer.

### Opportunity Types
CRM opportunities have an `opportunity_type` field that maps to the entity being sold to:

- `mso` вҖ” Prospective MSO (Master System Operator)

### Sales Cycles by Type
- Master Agent: Lead вҶ’ Opp вҶ’ NDA Sent вҶ’ NDA Signed вҶ’ Agreement Sent вҶ’ Negotiation вҶ’ Signed & Approved
- MSO: Lead вҶ’ Opp вҶ’ Agreement Sent вҶ’ Negotiation вҶ’ Signed & Approved
- Dealer: Lead вҶ’ Opp вҶ’ Agreement Sent вҶ’ Signed & Approved
- Install/Service Partner: Lead вҶ’ Vetting вҶ’ Agreement Sent вҶ’ Approved
- Sales Partner: Lead вҶ’ Opp вҶ’ Commission Agreement вҶ’ Signed
- Property/Company: Lead вҶ’ Opp вҶ’ Site Survey вҶ’ Quote Sent вҶ’ Quote Approved вҶ’ Signed вҶ’ Install Scheduled
- Customer: Lead вҶ’ Opp вҶ’ Quote Sent вҶ’ Quote Approved

### Onboarding Trigger ("Approve & Send Welcome")
Fires when opportunity = Won AND all required documents signed. Sends via Resend:
1. Welcome email with manual PDF link
2. Clerk portal invite (scoped to correct org tier)
3. Tier-specific welcome content (commission doc for MA, tech tool code for dealers, etc.)

For Property/Company wins: creates work order + site record + sends PM portal invite.

### Document Strategy
- NDA-A (Partner): Master Agent, MSO вҖ” full mutual NDA
- NDA-B (Dealer): Dealer, Install Partner, Service Partner вҖ” stronger one-way, protects dealer client lists
- NDA-C (Sales): Sales Partner вҖ” lightweight, protects leads/pricing only
- Agreements: One per tier (MA Agreement, MSO Agreement, Dealer Agreement, Install Partner Agreement, Service Partner Agreement, Sales Partner Agreement)
Install/Service Partner agreements have strongest non-solicitation clauses (they access client sites).

- Phase 1 (current): MSO display rename + opportunity_type field + type-based stage checklists

- Phase 3: "Approve & Send Welcome" trigger вҶ’ org creation + Resend welcome + Clerk invite

## Manual Upload Pipeline вҖ” LIVE вң… (May 2026) вҖ” Fully Automatic

**The pipeline is now zero-friction. Three paths, all end at the same place:**

### Path A вҖ” Find Online (one click, recommended)
1. Add product to `/products` with field_service=true
2. Click **Find Online** button вҶ’ `POST /api/kb/find-manual`
3. System searches known manufacturer URL patterns (DoorKing, Brivo, LiftMaster, Ubiquiti, etc.) then Claude URL lookup
4. PDF downloaded вҶ’ uploaded to Supabase Storage вҶ’ chunked + embedded вҶ’ `products.manual_url` updated
5. Wiring extraction auto-triggers in background вҶ’ terminal map in `device_suggestions`

### Path B вҖ” Upload PDF (manual file, any size)
1. Click **Upload PDF** вҶ’ file picker вҶ’ select PDF (up to 100 MB)
2. Step 1: browser calls `POST /api/kb/upload-url` вҶ’ gets Supabase signed URL (tiny JSON, no Vercel size limit)
3. Step 2: browser PUTs file **directly to Supabase Storage** вҖ” binary never touches Vercel
4. Step 3: browser calls `POST /api/kb/process` with `{ product_id, manual_url }` вҶ’ chunks + embeds
5. Wiring extraction auto-triggers in background

### Path C вҖ” Bulk script
- `scripts/bulk-upload-manuals.mjs` вҖ” batch process multiple products

**After any path:**
- `products.manual_url` is set вҶ’ green dot on /tech device picker
- `manual_chunks` populated вҶ’ AI diagnostic cites manual pages
- `device_suggestions` populated вҶ’ wiring guide shows AI terminal map for new devices
- OpenAI billing required вҖ” platform.openai.com/settings/billing

## Smart Tester API Integration (Future вҖ” /tech)

Field techs use Fluke, IDEAL, and Greenlee cable/network testers daily. Future goal: pull test results directly into the diagnostic session instead of requiring manual entry.

**Target integrations:**
- **Fluke LINKWARE Live** вҖ” Fluke's cloud platform for DSX/DTX cable analyzer results. REST API available. Test results (wiremap, insertion loss, NEXT, return loss) linkable per cable ID. вҶ’ Auto-populate measure steps with actual tester readings.
- **IDEAL AnyWARE Cloud** вҖ” IDEAL Networks tester cloud. Job-based test result storage with web API. Supports Cat5e/Cat6/fiber certification results.
- **Greenlee GT-8 / DataScout** вҖ” Bluetooth LE export from handheld testers. Some models support direct data transfer via USB or mobile app.
- **Generic multimeter (short-term)** вҖ” No API needed. Let tech photograph the multimeter display вҶ’ Claude vision extracts the reading and validates it against the expected range for that measure step. Same `/api/kb/analyze-image` endpoint already built.

**Implementation approach:**
1. Add `tester_brand` + `tester_job_id` optional fields to the `/tech` symptom screen
2. On measure steps, show a "Import from tester" button if credentials are configured
3. Dealer configures tester API key in `/settings` вҶ’ stored per-org in Supabase
4. Results auto-fill measure step answer + flag pass/fail vs expected range
5. Store tester report URL in `troubleshoot_sessions` for audit trail

**Immediate workaround (already works today):** Techs can photograph the tester screen on any photo step вҖ” Claude vision reads the display and comments on the reading.
